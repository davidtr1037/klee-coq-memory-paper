\section{Program Safety via Execution Trees}
\label{sec:program-safety}

In this section,
we present a theoretical foundation that enables us to prove safety of programs based on properties of the symbolic semantics.
The main result of this section is~\Cref{theorem:safety-via-et},
which states that a program is safe with respect to the concrete semantics if the execution tree of the program satisfies certian safety properties.

\subsection{Reachability Completeness}
\label{sec:completeness}

In this section,
we establish the \emph{reachability completeness} of the symbolic semantics with respect to the concrete semantics.
More specifically, our main result is~\Cref{theorem:completeness},
which states that any sequence of derivations in the concrete semantics that does not introduce $\tpoison$ values can be simulated by a corresponding sequence of derivations in the symbolic semantics.
Ultimately, this correspondence will help us proving the main theorem of~\Cref{sec:program-safety} (\Cref{theorem:safety-via-et}).

To establish reachability completeness,
we first define a relation between concrete and symbolic states.
Informally, we say that a symbolic state $s$ \emph{represents} a concrete state $c$ if $c$ can be obtained from $s$ via concretization using a model that satisfies the path constraints of $s$.
The formal definition is given below.

% TODO: same number of memory objects
\begin{definition}
\label{def:over-approx-state}
Let $m$ be a model.
An SMT expression $e$ \emph{represents} a value $v \in \Vals$ via $m$,
denoted by $e \overapproxvia{m} v$, if $\evalmodel{m}{e} = v$.
A symbolic memory object $(\addr, \size, \smtarray)$ represents a memory object $(\addr, \size, \arrayvalue)$ via $m$ if
$\arrayselect{\smtarray}{\bvconst{\indexwidth}{k}}$ represents $\arrayvalue(\valoftype{k}{\indextype})$ via $m$ for every $k$ in the range of the index type.
A symbolic heap $\symmemory$ represents a heap $\memory$ via $m$, denoted by $\symmemory \overapproxvia{m} \memory$,
if the symbolic memory objects of $\symmemory$ represent the memory objects of $\memory$ via $m$.
A symbolic state $s \triangleq \symstate{\location}{\symlocalstore}{\symstack}{\symmemory}{\pc}$ \emph{represents} a concrete state $c \triangleq \progstate{\location}{\localstore}{\stack}{\memory}$ via $m$,
denoted by $s \overapproxvia{m} c$,
if $\symlocalstore \overapproxvia{m} \localstore$,
$\symmemory \overapproxvia{m} \memory$,
and $\evalmodel{m}{\pc} = \true$.
Furthermore, $s$ \emph{represents} $c$, denoted by $s \overapprox c$,
if $s \overapproxvia{m} c$ for some model $m$.
\end{definition}

Based on~\Cref{def:over-approx-state},
one can see that a concrete state containing a $\tpoison$ value cannot be represented by a symbolic state.
This implies that we cannot establish the reachability completeness of the symbolic semantics with respect to the concrete semantics.
%Therefore, a sequence of derivations in the concrete semantics involving $\tpoison$ values cannot be simulated by a sequence of derivations in the symbolic semantics.
Instead, we establish the reachability completeness with respect to a subset of the concrete semantics restricted to concrete states that do not contain $\tpoison$ values.
%Formally, it is defined as follows:

In the following,
we formally define the restricted concrete semantics.
We first define \emph{poison-free} states, and then define the corresponding step relation.

\begin{definition}
A concrete state $c \triangleq \progstate{\location}{\localstore}{\stack}{\memory}$ is \emph{poison-free}
if $\localstore$ and $\memory$ do not contain $\tpoison$ values.
\end{definition}

\begin{definition}
We say that there is a \emph{poison-free} step between two concrete states $c$ and $c'$, denoted by $c \nsstep c'$,
if $c \step c'$ and both $c$ and $c'$ are poison-free.
\end{definition}

Similarly to~\Cref{def:safety},
we define when a program, \ie an \llvm module, is considered safe with respect to the restricted concrete semantics.
The formal definition is given below.

\begin{definition}
\label{def:issafe-ns-step}
Let $c_0 \triangleq \initstate(\module)$,
then $\module$ is \emph{safe} with respect to the restricted concrete semantics, denoted by $\issafens(\module)$, if:
\[ \forall c. \ c_{0} \multinsstep c \implies \neg \errorstate(c) \]
(The relation $\multinsstep$ is the reflexive-transitive closure of the relation $\nsstep$.)
\end{definition}

\begin{lemma}
\label{lemma:completeness-step}
Let $c$ be a concrete state and $s$ be a symbolic state.
If $s \overapprox c$ and $c \nsstep c'$,
then there exists a symbolic state $s'$ such that $s \symstep s'$ and $s' \overapprox c'$.
\end{lemma}

Now, the reachability completeness theorem is given below.

\begin{theorem}
\label{theorem:completeness}
Let $c_{0} \triangleq \initstate(\module)$, and let $s_{0} \triangleq \initsymstate(\module)$.
If $\module$ is supported and $c_{0} \multinsstep c$,
then there exists a symbolic state $s$ such that $s_{0} \symmultistep s$ and $s \overapprox c$.
\end{theorem}

An execution tree is a data structure that records the paths explored during the analysis of a given program.
We denote by $\tree(s, l)$ the execution tree $t$ in which $s$ is the root symbolic state ($t.\attrstate$) and $l$ is a list of successor subtrees ($t.\attrchildren$).
Following the approach of~\cite{klee-rocq},
we show that program safety can be proved based on certain safety properties of execution trees.
To accommodate memory-related instructions,
we extend the relevant definitions, lemmas, and theorems.

In general, the symbolic states derived by the symbolic semantics may differ from those computed by the \SE engine.
To relate between them,
we extend the definition of \emph{equivalence} for symbolic states that was given in~\cite{klee-rocq} (Definition 3.12) to account for memory.

Now, the safety property of execution trees is defined inductively below.
\begin{definition}
\label{def:safe-et}
Let $s$ be a symbolic state and $l$ a list of execution trees.
We say that $tree(s, l)$ is safe, denoted by $\safeet(\tree(s, l))$,
if (1) $s$ is not an error state,
and (2) if $s \symstep s'$ then either the path constraints of $s'$ are unsatisfiable or there exists a subtree $t \in l$ such that $\safeet(t)$ and $s' \equivstate t.\attrstate$.

This property ensures that there are no error states in the tree, and furthermore,
that every feasible successor $s'$ of any state $s$ occurring in the tree is also represented in the tree as a successor of $s$.
Infeasible states can be ignored, as they do not represent any concrete states.

The following theorem states that if the execution tree of the program satisfies the safety property described above,
then the program is safe w.r.t. the concrete semantics.

\begin{theorem}
\label{theorem:safety-via-et}
Let $s_0 \triangleq \initsymstate(\module)$, and $l$ be a list of execution trees.
If $\module$ is supported and $\safeet(\tree(s_0, l))$, then $\issafe(\module)$.
\end{theorem}
