\section{Overview}
\label{sec:overview}

% The main insight behind the technique is that:

In this section,
we give an informal overview of our approach before moving on to the formal development (\Cref{sec:program-safety}).
For the purpose of this overview,
we assume a standard \emph{mechanized} and \emph{executable} symbolic semantics, without relying on specific implementations.
The \emph{mechanized} symbolic semantics are those explicitly encoded inside the proof assistant (\eg \coq).
The \emph{executable} symbolic semantics are those implicitly induced by the implementation of the \SE engine (\eg \klee).
In general,
those semantics are not necessarily identical,
since \SE engines employs various encoding optimizations which may not be fully modeled by mechanized semantics.

\subsection{Motivating Example}

Consider the \llvm IR program shwon on the left side of~\Cref{figure:overview-llvm}.
Suppose that we want to prove that this program is \emph{safe},
\ie it does not trigger errors for any valid inputs.
The program first allocates a heap object (\cref{line:overview-llvm-malloc}) and then,
if a certain condition over \code{n} is satisfied (\cref{line:overview-llvm-br}), it accesses the allocated object at offset \code{n} (\cref{line:overview-llvm-store}).

% what happens with a standard SE engine?
A standard \SE engine analyzes this program by executing it with symbolic values rather than concrete ones.
It performs the analysis by tracking symbolic states that abstractly represent the program environment.
Each symbolic state consists of a program counter, a symbolic store for registers, a symbolic memory, and path constraints accumulated along the execution path.

An \emph{execution tree} is a data structure that describes the paths explored during the analysis.
In this example,
the execution tree of the program is shown on the right side of~\Cref{figure:overview-llvm}.
More specifically,
it consists of the yellow nodes which correspond to the symbolic states derived by the executable symbolic semantics, \ie those computed by the \SE engine.\footnote{
For now, ignore the blue nodes.
}

In this example,
the \SE engine models the input \code{n} as a symbolic value $n$.
It starts the symbolic execution of the program from the initial state $s_0$.
At~\cref{line:overview-llvm-malloc},
it allocates a heap object of size 10, resulting in the symbolic state $s_1$.
At~\cref{line:overview-llvm-br},
both sides of the branch are feasible,
so it forks the state $s_2$ into two states: $s_3$ and $s_4$.
At~\cref{line:overview-llvm-store},
it stores the value \code{7} into the address given by \code{p}, resulting in the symbolic state $s_6$.
To do so,
it validates that this access may not be out-of-bounds by querying the SMT solver to check the satisfiability of the condition $n / 2 = 4 \land \neg (0 \leq n < 10)$.
In this case, this condition is unsatisfiable, so this access is guaranteed to be error-free.
Overall, it explores all the feasible paths without reporting any errors.

% why is it not enoguh?
However, a standard \SE engine is a complex software system.
A single bug in its implementation can compromise the integrity of the analysis, leading to undetected errors.
To reduce the reliance on a massive trusted computing base,
our goal is to generate an independent and machine-checked proof ensuring that an exhaustive exploration of the input program indeed guarantees the safety of that program.

% theory
\subsection{Theoretical Framework}

To generate machine-checked safety proofs, we develop a formal framework in \coq.
Our theoretical framework defines the concrete (\Cref{sec:concrete-semantics}) and symbolic (\Cref{sec:symbolic-semantics}) semantics of \llvm.
%When the symbolic execution of a given program is \emph{exhaustive},
%the explored paths form an \emph{execution tree},
%where each node represents a symbolic state computed by the \SE engine.
The main result of our theoretical framework is the \emph{program safety} theorem (\Cref{theorem:safety-via-et}).
Intuitively, this theorem states that if the execution tree of the program satisfies certain \emph{safety properties},
then the program is safe with respect to the concrete \llvm semantics.
More specifically, the required safety properties are \emph{error-freeness} and \emph{completeness} (\Cref{def:safe-et}).
The former requires that the execution tree must not contain error symbolic states.
The latter requires that the execution tree over-approximates all the feasible symbolic states derivable by the symbolic \llvm semantics.
This theorem bridges the concrete and symbolic domains,
providing a foundation for automating proof generation.

% proof generation
\subsection{Proof Generation}

To apply the \emph{program safety} theorem,
we must provide an execution tree that satisfies the safety properties mentioned above.
To do so, we instrument the \SE engine to generate an execution tree along with a proof showing that it satisfies the desired safety properties.
More specifically,
for every instruction executed by the \SE engine,
we generate a lemma that proves the safety properties of the corresponding subtree.
This way, we decompose the global proof obligation of the entire program into instruction-level proof obligations.

To illustrate the proof generation process, consider again the \llvm IR program in~\Cref{figure:overview-llvm}.
On the right side of~\Cref{figure:overview-llvm},
the blue nodes correspond to the symbolic states derived by the mechanized symbolic semantics.
Recall that the yellow nodes correspond to the symbolic states derived by the executable symbolic semantics.
%Each node $s_i$ corresponds to a symbolic state,
%and its successors are the symbolic states computed by the \SE engine while executing the current instruction of $s_i$.
Here, for example,
$s_1$ is the symbolic state derived from $s_0$ by the executable symbolic semantics,
\ie $s_1$ is computed by the \SE engine when executing the instruction at~\cref{line:overview-llvm-malloc},
and $\bar{s}_1$ is the symbolic state derived from $s_0$ by the mechanized symbolic semantics.

When the \SE engine executes the current instruction of $s_i$,
we generate a lemma stating that the subtree of $s_i$ satisfies the safety properties.
Once the analysis of the whole program is complete,
we compose those lemmas in a bottom-up manner to obtain a lemma stating that the whole execution tree satisfies the safety properties.
Here, for example,
the lemmas corresponding to $s_3$ and $s_7$ are generated first, and the lemma corresponding to $s_0$ is generated only at the end.

To give intuition about how those lemmas are proven, consider the subtree of $s_5$.\footnote{
An extended and formal example will be given in~\Cref{sec:proof-generation}.
}
The current instruction of $s_5$ is the \code{store} instruction at~\cref{line:overview-llvm-store}.

First, we need to prove that this subtree is \emph{error-free}.
More specifically,
we need to prove that $s_5$ is not an error symbolic state.
In this case,
we need to show that the address given by \code{p} cannot point to an out-of-bounds address.
Here, this address points to a single object,
the one allocated at~\cref{line:overview-llvm-malloc},
so it will suffice to prove that the constraint $n / 2 = 4 \land \neg (0 \leq n < 10)$ is unsatisfiable.

Second, we need to prove that this subtree is \emph{complete}.
More specifically,
we need to prove that if $\bar{s}$ can be derived from $s_5$ by the mechanized symbolic semantics,
then either $\bar{s}$ is infeasible, \ie its path constraints are unsatisfiable, or there exists a successor of $s_5$ in the subtree that is equivalent to $\bar{s}$.
In the case of a \code{store} instruction,
the mechanized symbolic semantics may derive multiple symbolic states,
depending on the number of objects that may be pointed by the accessed address.
Here, the address given by \code{p} points to a single object, so there is only one derived symbolic state, namely $\bar{s}_6$.
This state is not infeasible,
so we need to prove that $s_6$ and $\bar{s}_6$ are equivalent.
%In general, the mechanized semantics may differ from the symbolic semantics of the \SE engine,
%so the equivalence proof might not be immediate.

Lastly, we need to prove that the subtree of $s_6$, \ie the successor of $s_5$, satisfies the safety properties as well.
This follows immediately since the proof is constructed in a bottom-up fashion.
