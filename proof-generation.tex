\section{Proof Generation}
\label{sec:proof-generation}

Ultimately, our goal is to prove that the analyzed program is safe.
%\Cref{theorem:safety-via-et} implies that the existence of a safe execution tree guarantees program safety with respect to the concrete semantics.
%This result is crucial, as it provides the foundation for constructing safety proofs.
\Cref{theorem:safety-via-et} guarantees that the program is safe with respect to the concrete semantics if one proves that the execution tree of that program is safe.
In this section, we show how to compute such an execution tree and how to prove its safety properties.
%Our approach works as an instrumentation of the \SE engine, and we extend it to support memory operations.

The final outcome of our approach is a \coq proof consisting of a \emph{static} and \emph{dynamic} sections.
The static section is written manually and it contains the various definitions that were given in~\Cref{sec:concrete-semantics,sec:symbolic-semantics,sec:program-safety}:
concrete semantics, symbolic semantics, etc.
The dynamic section is generated by the \SE engine during the analysis of the program,
and it contains definitions that are specific to that program:
module definition, symbolic states, execution tree, etc.
First, we generate the module definition and the proof of the module assumptions.
During the symbolic execution,
we generate the definitions of the symbolic states and the execution tree.
For each instruction executed by the \SE engine,
we generate a lemma that proves the safety property (\Cref{def:safe-et}) for the corresponding subtree in the execution tree.
After we generate the proof showing that the whole execution tree satisfies that safety property, we apply~\Cref{theorem:safety-via-et}.

Here, we present our approach using \klee~\cite{klee}, a \emph{state-of-the-art} \SE engine.
% TODO: improve
However, we note that our approach can be adapted to other \SE implementations as well.
Before diving into the details,
we give some background about the internals of the memory modeling in \klee.

\subsection{Memory Modeling in \klee}
%In this section,
%we give some background about the memory modeling in \klee.

% TODO: unique interval?
In \klee, the memory of a symbolic state is represented as a linear address space containing memory objects,
each associated with a concrete base address and a concrete size.
To encode the contents of a memory object,
\klee uses two representations: \emph{optimized} and \emph{non-optimized}.
If all the stores to a memory object were performed with a concrete offset,
then \klee uses the optimized representation:
it encodes the contents of that memory object as a mapping between constant offsets and symbolic values.
Otherwise, if there is at least one store with a symbolic offset,
\klee uses to the non-optimized representation:
it translates the optimized representation into an SMT array, \ie a sequence of store expressions.
Once the representation of a memory object is switched to the non-optimized one, it cannot be switched back to the optimized one.

%The SMT arrays that are used to encode the contents of a memory object has the sort: ...

For example, after executing~\cref{line:example-store} of the C program in~\Cref{figure:example-llvm},
the contents of the memory object allocated at~\cref{line:example-malloc} are encoded using the optimized representation as:
\[ \{ 0 \mapsto \bvconst{8}{7}, 1 \mapsto \bvconst{8}{0} \} \]
If afterwards, we were to execute an instruction that writes the value \code{1} to the address \code{p + n},
then we would switch to the non-optimized representation, resulting in the following encoding:
\[ \arraystore{\arraystore{\arraystore{\arrayconst{\arrayinitvalue}}{\bvconst{32}{0}}{\bvconst{8}{7}}}{\bvconst{32}{1}}{\bvconst{8}{0}}}{n}{\bvconst{8}{1}} \]

\subsection{Translating Modules and Symbolic States}
\label{sec:proof-translation}
We translate the \llvm module into its corresponding representation in \coq,
and generate a proof showing that the module assumptions are indeed satisfied (\Cref{sec:module-assumptions}).
During the analysis, we translate the symbolic states generated by the \SE engine into their corresponding representation in \coq based on~\Cref{def:symbolic-state}.
We translate memory objects from \klee's internal representation into \coq as follows.
In the case of an optimized representation,
we translate the internal mapping $\{0 \mapsto e_0, 1 \mapsto e_1, \ldots\}$ into a sequence of store expressions in \coq:
\[ \arraystore{\arraystore{\arrayconst{\arrayinitvalue}}{\bvconst{32}{0}}{e_0}}{\bvconst{32}{1}}{e_1} \ldots \]
In the case of a non-optimized representation,
we directly translate the internal sequence of store expressions into a sequence of store expressions in \coq.
Accordingly, we translate the internal heap into a list of memory objects in \coq.

\subsection{Proving Execution Tree Safety using Interface Lemmas}
\label{sec:proof-et}

The most challenging part in the proof generation process is proving that the execution tree of the program is safe.
This eventually enables us to prove program safety using~\Cref{theorem:safety-via-et}.
In this section, we describe in detail how to generate this part of the proof.

First, we generate the definition of the execution tree in a bottom-up manner.
Then, we have to prove that the resulting execution tree satisfies the $\safeet$ relation (\Cref{def:safe-et}).
We generate this proof in a bottom-up manner as well, showing that each subtree satisfies the required relation.
The proof of each subtree corresponds to a single instruction executed during the analysis.
Generating explicit proofs from scratch for each subtree is expensive in terms of proof generation,
and it also makes the proof checking inefficient,
as such proofs require expensive \coq tactics (\eg \textit{inversion} and \textit{subst}).
Instead, we create for each instruction type a generic lemma that can be used to infer execution tree safety.
We call these lemmas \emph{interface} lemmas,
as they facilitate an effective interaction between the static and dynamic sections of the proof.
By using such interface lemmas,
we accelerate both proof generation and proof checking,
since we create smaller proofs which mainly rely on simple \coq tactics (\eg \textit{apply} and \textit{reflexivity}).

Let $s_0 \triangleq \symstate{\location_1}{\{ \}}{[]}{[]}{\true}$ by the initial symbolic state at~\cref{line:llvm-malloc},
and suppose that we want to prove that the subtree of $s_0$ is safe.

In general, the \rulenameref{Malloc} rule (\Cref{sec:symbolic-semantics}) may derive multiple symbolic states.
Here, however, the size argument of \textit{malloc} is evaluated to a constant SMT expression,
so the only derived symbolic state is:
\[ \mech{s}_1 \triangleq \symstate{\location_2}{\{ \nameid{p} \mapsto \bvconst{64}{\addr} \}}{[]}{[(\bvval{\addr}{64}, \bvval{2}{32}, \arrayconst{\arrayinitvalue})]}{\true} \]
%\[
%s' \triangleq \symstate
%    {\location_2}
%    {\{ \nameid{p} \mapsto \bvconst{64}{\addr} \}}
%    {[]}
%    {[(\bvval{\addr}{64}, \bvval{2}{32}, \arrayconst{\arrayinitvalue})]}
%    {\true}
%\]
where $\bvval{\addr}{64}$ is the allocated base address.
At this point, the \SE engine uses an optimized representation for the allocated memory object, as no stores have been performed yet,
so the symbolic state computed by the \SE engine is:
\[
s_1 \triangleq \symstate
    {\location_2}
    {\{ \nameid{p} \mapsto \bvconst{64}{\addr} \}}
    {[]}
    {[(\bvval{\addr}{64}, \bvval{2}{32}, \arraystore{\arraystore{\arrayconst{\arrayinitvalue}}{\bvconst{32}{0}}{\arrayinitvalue}}{\bvconst{32}{1}}{\arrayinitvalue})]}
    {\true}
\]
Now, we have to prove that the subtree of $s_0$ is safe.
Based on~\Cref{def:sym-error-state},
$s_0$ is not an error state since $\evalexprty{2}{\symlocalstore}{\bvsort{64}} = \bvconst{64}{2}$.
To prove that $\mech{s}_1 \equiv s_1$,
it suffices to prove that their memory is equivalent.
The SMT arrays in those two symbolic states are encoded differently,
but they are actually equivalent, since the additional store expressions contain the same value $\arrayinitvalue$ as in $\arrayconst{\arrayinitvalue}$.

To handle such cases in general, we define an interface lemma for \textit{malloc}, which is stated as~\Cref{lemma:interface-malloc}.
In this case, we can apply~\Cref{lemma:interface-malloc} with $n$ as 2, $\mo$ as the memory object from $s_1$, and $t$ as the subtree of $s_1$.
% TODO: show why the preconditions hold?

\begin{lemma}
\label{lemma:interface-malloc}
Let $s \triangleq \symstate{\location}{\symlocalstore}{\symstack}{\symmemory}{\pc}$,
where \textup{$\instrat{\location} = \codequote{x~\texttt{=}~\tcall^{\pointertypeof{\bvsort{8}}}~\intrinsicfunc{malloc}(\bvsort{64} ~ e)}$},
and let $\bvval{\addr}{64}$ be the allocated base address.
If there exist $n$, $\mo$, and $t$ such that:
\begin{enumerate}
    \item \textup{$\evalexprty{e}{\symlocalstore}{\bvsort{64}} \equiv \bvconst{64}{n}$} and \textup{$\bvval{n}{64} < \bvval{(2^{\indexwidth - 1} - 1)}{64}$}
    \item \textup{$\mo \equiv (\bvval{\addr}{64}, \bvval{n}{32}, \arrayconst{\arrayinitvalue})$}
    \item \textup{$t.\attrstate = \symstate{\nextinstr{\location}}{\symlocalstore[x \mapsto \bvconst{64}{\addr}]}{\symstack}{\symmemory \cup \{\mo\}}{\pc}$}
    \item $\safeet(t)$
\end{enumerate}
then $\safeet(\tree(s, [t]))$.
\end{lemma}

Note that this lemma assumes that the size argument of \textit{malloc} is evaluated to a concrete SMT expression.
If that is not the case,
then the \SE engine typically concretizes the symbolic expression,
and then the analysis may be incomplete and a safety proof may not be derivable.
There exist \SE engines that support symbolic-size allocations (\eg \cite{symsize-model}),
however, it is not the focus of this work.

Consider the symbolic state before the execution of~\cref{line:llvm-store}:

s_5 \triangleq \symstate
    {\location_6}
    {\{ ..., \nameid{gep1} \mapsto \bvconst{64}{\addr} \}}
    {[]}
    {[\mo]}
    {n < \bvconst{32}{2}}
\]
where:
\[ \mo \triangleq (\bvval{\addr}{64}, \bvval{2}{32}, \arrayvalue), \ \arrayvalue \triangleq \arraystore{\arraystore{\arrayconst{\arrayinitvalue}}{\bvconst{32}{0}}{\arrayinitvalue}}{\bvconst{32}{1}}{\arrayinitvalue} \]

According to the symbolic semantics,
if $\mech{s}$ is derived from $s_5$ then there exist a model $m$ and a memory object $\mo'$ such that
$m \models (n < \bvconst{32}{2})$, $\mo' \in \symmemory$, and $\symisinbounds{\bvconst{64}{\addr}}{\bvsort{8}}{\mo'}{m}$, and then:
\[
\mech{s}_6 \triangleq \symstate
    {\location_7}
    {\{ ..., \nameid{gep1} \mapsto \bvconst{64}{\addr} \}}
    {[]}
    {[\mowrite{\bvsort{8}}{\mo'}{\bvconst{64}{\addr}}{\bvconst{8}{7}}]}
    {n < \bvconst{32}{2} \land \rangecond(\bvconst{64}{\addr}, 1, \mo'))}
\]
Recall that we assume that address intervals of memory objects do not overlap.
Since $m[\bvconst{64}{\addr}]$ is the base address of both $\mo$ and $\mo'$, and $\bvconst{64}{\addr}$ is a constant SMT expression,
then it must hold that $\mo = \mo'$.
Therefore, although the \rulenameref{Store} rule (\Cref{sec:symbolic-semantics}) may have multiple derivations in general,
there is a single derivation in this case.

In particular, it also implies that:
\[ \mowrite{\smtsort{8}}{\mo'}{\bvconst{64}{\addr}}{\bvconst{8}{7}} = (\bvval{\addr}{64}, \bvval{2}{32}, \arraystore{\arrayvalue}{(\bvextract{32}{\bvsub{\bvsort{64}}{\bvconst{64}{\addr}}{\bvconst{64}{\addr}}})}{\bvconst{8}{7}}) \]

The symbolic state computed by the \SE engine is:
\[
s_6 \triangleq \symstate

    {[]}
    {[\mo_{new}]}

\]
where:
\[ \mo_{new} \triangleq (\bvval{\addr}{64}, \bvval{2}{32}, \arrayvalue_{new}), \ \arrayvalue_{new} \triangleq \arraystore{\arraystore{\arrayconst{\arrayinitvalue}}{\bvconst{32}{0}}{\bvconst{8}{7}}}{\bvconst{32}{1}}{\arrayinitvalue} \]
Now, we have to prove that the subtree of $s_5$ is safe.

Note that $s_5$ is not an error state,
since $\evalexprty{\nameid{gep1}}{\symlocalstore}{\bvsort{64}} = \bvconst{64}{\addr}$ for every model $m$.
To prove that $\mech{s}_6 \equiv s_6$, it suffices to prove the equivalence of their memory and path constraints.
As for the memory, it suffices to prove that:
\[ \arraystore{\arrayvalue}{(\bvextract{32}{\bvconst{64}{\addr} - \bvconst{64}{\addr}})}{\bvconst{8}{7}} \equiv \arrayvalue_{new} \]

\begin{lemma}
Let $\bvconst{32}{i}$, ..., $\bvconst{32}{(i + n)}$ be subsequent constant SMT expressions,
and assume that $k \equiv \bvconst{32}{i}$ and $v' \equiv v_{new}$, then:
\[ \arraystore{\arraystore{\arraystore{a}{\bvconst{32}{i}}{v}[...]}{\bvconst{32}{(i + n)}}{t_n}}{k}{v'} \equiv \arraystore{\arraystore{a}{\bvconst{32}{i}}{v_{new}}[...]}{\bvconst{32}{(i + n)}}{t_n} \]
\end{lemma}

This lemma can be applied in our case,
since $\bvextract{32}{\bvsub{\bvsort{64}}{\bvconst{64}{\addr}}{\bvconst{64}{\addr}}} \equiv \bvconst{32}{0}$ and $\bvconst{8}{7} \equiv \bvconst{8}{7}$.

As for the path constraints,
note that $\rangecond(\bvconst{64}{\addr}, 1, \mo) \equiv \true$, so the equivalence follows easily.

To handle such cases in general, we define an interface lemma for \textit{store}, which is stated as~\Cref{lemma:interface-store}.

\begin{lemma}
\label{lemma:interface-store}
Let $s \triangleq \symstate{\location}{\symlocalstore}{\symstack}{\symmemory}{\pc}$,
where \textup{$\instrat{\location} = \codequote{\tstore^{\tau}~e_1~e_2}$}.
If there exist $\mo$, $\mo'_{opt}$, and $t$ such that:
\begin{enumerate}
    \item $\symmemory$ is well-formed
    \item $\mo \in \symmemory$
    \item $\evalexprty{e_1}{\symlocalstore}{\tau} = v$
    \item $\evalexprty{e_2}{\symlocalstore}{\bvsort{64}} = \bvconst{64}{p}$
    \item $\isinbounds{\bvval{p}{64}}{|\tau|}{\mo}$
    \item $\mo'_{opt} \equiv \mowrite{\tau}{\mo}{\bvconst{64}{p}}{v}$
    \item $t.\attrstate = \symstate{\nextinstr{\location}}{\symlocalstore}{\symstack}{\heapreplace{\symmemory}{\mo}{\mo'_{opt}}}{\pc}$
    \item $\safeet(t)$
\end{enumerate}
then $\safeet(\tree(s, [t]))$.
\end{lemma}

\subsubsection{Example: Load}
% TODO: remind that the repr is non-opt now
The symbolic state at~\cref{line:llvm-load} is:
\[
s \triangleq \symstate
    {\location_9}
    {\{ ..., \nameid{gep2} \mapsto e \}}
    {[]}
    {[\mo]}
    {n < \bvconst{32}{2}}
\]
where:
\[ e \triangleq \bvadd{\bvsort{64}}{\bvconst{64}{\addr}}{\bvzext{64}{\bvextract{32}{\bvzext{64}{n}}}} \]
\[ \mo \triangleq (\bvval{\addr}{64}, \bvval{2}{32}, \arrayvalue), \ \arrayvalue \triangleq \arraystore{\arraystore{\arrayconst{\arrayinitvalue}}{\bvconst{32}{0}}{\bvconst{8}{7}}}{\bvconst{32}{1}}{\arrayinitvalue} \]

On one side, according to the symbolic semantics,
if $s'$ is derived from $s$ then there exist a model $m$ and a memory object $\mo' \triangleq (\addr', \size', \arrayvalue')$ such that
$m \models \pc$, $\mo' \in \symmemory$, and $\symisinbounds{e}{\bvsort{8}}{\mo'}{m}$, and then:
% TODO: should add 0_{32}?
\[
s' \triangleq \symstate
    {\location_{10}}
    {\{ ..., \nameid{gep2} \mapsto e, \nameid{v} \mapsto \arrayselect{\arrayvalue'}{\bvextract{32}{\bvsub{\bvsort{64}}{e}{\bvconst{64}{\addr}}}} \}}
    {[]}
    {[\mo]}
    {n < \bvconst{32}{2} \land \rangecond(e, 1, \mo')}
\]
Before dereferencing $e$, the \SE engine checks if $e$ may point to other memory objects,
by checking the satisfiability of $n < \bvconst{32}{2} \land \neg \rangecond(e, 1, \mo)$.
Here, this constraint is unsatisfiable,
which means that $e$ points only to $\mo$.
In particular, it must hold that $\mo' = \mo$ (and $\arrayvalue' = \arrayvalue$).

One the other side, the symbolic state computed by the \SE engine is:
\[
s'_{opt} \triangleq \symstate
    {\location_{10}}
    {\{ ..., \nameid{gep2} \mapsto e, \nameid{v} \mapsto \arrayselect{\arrayvalue}{\bvextract{32}{\bvsub{\bvsort{64}}{e}{\bvconst{64}{\addr}}}} \}}
    {[]}
    {[\mo]}
    {n < \bvconst{32}{2}}
\]
Now, we have to prove that the subtree of $s$ is safe.
We already know that $n < \bvconst{32}{2} \land \neg \rangecond(e, 1, \mo)$ is unsatisfiable,
so by~\Cref{def:sym-error-state}, $s$ is not an error state.
To prove that $s' \equiv s'_{opt}$, it suffices to prove the equivalence of their path constraints.
Since $n < \bvconst{32}{2} \land \neg \rangecond(e, 1, \mo)$ is unsatisfiable,
this implies that $n < \bvconst{32}{2} \land \rangecond(e, 1, \mo) \equiv n < \bvconst{32}{2}$.
